# Plan: After-call summary · Profanity filter · Barge-in (2026-07-10)

Investigation done via 3 read-only agents. Implement **one at a time**. All file:line refs verified
during investigation (may drift as code changes).

---

## 1) After-call summary — is it dynamic or fixed?

**Verdict: ALREADY DYNAMIC on the happy path, with silent static fallbacks.** Not "fixed by design".

- **Dynamic (real, per-conversation):** the 4 card *values* — mood / focus topic / energy shift /
  next step — are generated by **GPT-4o over the actual turn-by-turn transcript + emotion timeline**
  (`nowli-ai/test17.py` `chat_summary` `:925-966`, prompt `_build_summary_prompt` `:305-327`). The
  Flutter screen binds them at `call_summary_screen.dart:218/227/236/245`.
- **Static by design (labels/chrome):** headline "GREAT JOB!", the card *titles*
  ("Mood detected"…), icons, layout — always hardcoded. These are labels; fine as-is.
- **Silent fallbacks (the real issue):** the summary degrades to canned text WITHOUT telling the user:
  1. Backend: if the GPT call/parse fails → `_SUMMARY_FALLBACKS` canned text returned **as HTTP 200**
     (`test17.py:949-951, 283-302`) — looks real but isn't.
  2. Frontend: if `sessionId` missing / non-200 / 15s timeout → hardcoded `??` placeholders
     (`call_summary_screen.dart:34-59, 218-245`). Triggered e.g. when nowli-ai was unreachable during
     the call so no `sessionId` was passed (`ai_voice.dart:1010/1032`).

**Plan (small):**
- (a) **Verify** the dynamic path end-to-end in a live call (nowli-ai up + call has turns) → confirm real
  per-conversation text appears. Likely already works.
- (b) **Reliability hardening** so it doesn't silently fall back: ensure `sessionId` is always passed;
  confirm the in-memory nowli-ai session still exists at summary time (sessions are in-memory, lost on
  restart). Optionally a 1× retry on timeout.
- (c) **Decision needed:** keep the friendly static fallback (current), OR surface a subtle "couldn't
  generate this time" state, OR log/telemetry when fallback triggers so we know how often it happens.
- **Effort:** small. Mostly verification + optional hardening. No architecture change.

## 2) Profanity / inappropriate-content filter

**Current state: NONE anywhere** (nowli-ai and Django both confirmed clean). Only guard is an
empty-message check (`test17.py:891-892`).

**Recommended approach: OpenAI Moderation API (`omni-moderation-latest`).**
- Free, no new dependency/key (reuses the existing `AsyncOpenAI` client `_get_openai_client()`
  `test17.py:505-512`), multilingual (app supports many languages), far better than a static word list.
- **Server insertion point:** `chat_stream`, right after the empty-message guard and BEFORE emotion
  detection / model call / `session.add_turn` — `test17.py:893` (before `:895`/`:902`). A flagged
  message never reaches the model or the persisted history.
- **Client signal:** reuse the existing SSE event convention (`sse_event` `test17.py:70-72`; client
  already parses `emotion`/`word`/`done`/`error`). Emit a dedicated `warning` frame (cleaner) or reuse
  `error` (zero client change), skip the model call, close with `done`.
- **Client (ai_voice.dart):** handle the warning → show a notice (existing `NotificationManager`
  pattern) + optionally a gentle spoken TTS line ("Let's keep it kind — I'm here for you"), then resume
  listening. SSE parsing is in `ai_call_service.dart:116-224`; the sink is `_sendMessageToAi`
  (`ai_voice.dart:743-833`).

**⚠️ Critical design nuance (this is a MENTAL-HEALTH app):** do NOT block emotional distress /
self-harm / sadness. Moderation must target **profanity, harassment, hate, sexual content** — but a
distressed user saying they feel hopeless must get a **supportive** response, never a "you can't say
that" block. OpenAI moderation exposes `self-harm` categories separately, so we explicitly EXCLUDE
those from the block list and let the companion respond with care (ties into existing low-mood detect).

**Decisions needed:**
- (a) Confirm block categories = profanity/harassment/hate/sexual; EXCLUDE self-harm/distress. ✅ recommend.
- (b) Warn via visual notice only, or also a spoken TTS warning?
- (c) Escalation: warn every time (simple), or after N violations end the call? Recommend: warn only for now.
- **Effort:** medium. nowli-ai (server) + ai_voice.dart (client SSE handling + notice UI).

## 3) Barge-in (interrupt the AI mid-response for a fluid conversation)

**Current state:** STT is HARD-GATED OFF while the AI speaks — the mic never listens during TTS.
- Gate: `_startListening` returns early `if (_isSpeaking) return;` — `ai_voice.dart:511-514`.
- All auto-restart paths also blocked by `_isHandlingAiResponse` while a response streams.
- TTS is queue-driven (`_processTtsQueue` `:415-451`, `_isSpeaking` set `:431`/cleared `:417`); resumes
  listening only 1000ms AFTER the queue drains (anti-echo settle, `:419-427`).

**Plan (client-only, `ai_voice.dart`):**
1. **Let STT run during TTS:** relax the `if (_isSpeaking) return;` gate (`:511`) so the mic listens
   while the AI speaks.
2. **Detect real user speech during playback** via `onSoundLevelChange` (`:594-600`, RMS vs
   `_micSpeakingThreshold=2.0` `:33`) and/or `onResult` (`:560-591`).
3. **Barge-in teardown** (reuse the existing pattern at `:751-762`): `await _flutterTts.stop()` →
   `_ttsQueue.clear()` → `_isSpeaking=false` → cancel the in-flight SSE loop / reset
   `_isHandlingAiResponse` → then handle the new user input / `_startListening()`.

**⚠️ Main risk — echo / self-trigger:** the mic hearing the AI's own TTS and interrupting itself. The
current design AVOIDS this on purpose; barge-in removes that protection. Mitigations:
- Require a genuine energy threshold (raise `_micSpeakingThreshold`) + a minimum word count/confidence
  before treating it as barge-in (ignore short blips / TTS tail).
- Rely on device acoustic echo cancellation (AEC) — **works far better on a real phone than the
  emulator** (emulator mic routing is already flaky). **This feature must be validated on a physical
  device.**
- **Effort:** medium, but delicate + needs device testing. Do LAST.

## Recommended order
1. **Summary** (smallest — verify + optional hardening; quick win).
2. **Profanity filter** (well-scoped, high safety value; the self-harm nuance matters).
3. **Barge-in** (most delicate; needs a physical device to validate echo handling).

## PROGRESS
- **1) Summary — DONE (2026-07-10).** Verified the dynamic path live (session → 2 turns → `/chat/summary`
  returned real per-conversation text about the exact topics, ~2.1s). Confirmed **dynamic, not fixed**.
  Per decision "honest fallback only": replaced the fake-specific static fallbacks with honest ones that
  don't fabricate a mood/topic/arc — frontend `call_summary_screen.dart` (the `?? …` card defaults, shown
  when there's no real summary) and backend `nowli-ai/test17.py` `_SUMMARY_FALLBACKS` (en/de/es, shown when
  a real chat happened but GPT summarization failed). No retry added (per decision). `py_compile` OK,
  `flutter analyze` clean, both services restarted/rebuilt.
- **2) Profanity filter — DONE (2026-07-10).** Hybrid moderation in `nowli-ai/test17.py` `chat_stream`
  (after `_get_session`, before emotion/model/add_turn): (1) local `_PROFANITY_WORDS` list (EN + SR — the
  OpenAI Moderation API does NOT flag plain swear words), (2) OpenAI `omni-moderation-latest` for
  harassment/hate/sexual (`_BLOCK_MODERATION_CATEGORIES`). **Excludes self-harm/violence/distress**
  (wellness app). Fails open. On block → emits `warning` + `done` SSE (no `word`), skips model + turn
  persistence. Client: `warning` added to `StreamEventType` (`ai_call_models.dart`), parsed in
  `ai_call_service.dart`, handled in `ai_voice.dart` `_sendMessageToAi` → SnackBar notice + spoken TTS
  warning (`_MODERATION_WARNING` en/de/es), then resumes listening. **Verified live:** clean→reply,
  profanity→warning-only, distress ("hopeless/worthless")→full supportive reply (NOT blocked). `py_compile`
  OK, `flutter analyze` clean.
- **3) Barge-in — DONE (2026-07-10).** Client-only (`ai_voice.dart`). The mic now stays open while
  the AI speaks (removed the `if (_isSpeaking) return;` gate; `_processTtsQueue` starts listening when
  speech begins). On a real user phrase (>= `_bargeInMinWords` = 2) while `_isSpeaking`/streaming →
  `_interruptAiForBargeIn()` stops TTS + clears queue + breaks the SSE loop (`_bargeInterrupt`), then the
  final result sends normally. Feature flag `_bargeInEnabled`. `flutter analyze` clean. **Echo caveat:**
  emulator w/ speakers may self-trigger; validate on a physical phone (hardware AEC). Tuning knobs:
  `_bargeInMinWords`, `_bargeInEnabled`.
  - **UPDATE 2026-07-10 — DISABLED by default (`_bargeInEnabled = false`).** Emulator testing showed the
    open-mic-during-TTS approach breaks on devices without hardware AEC: the mic hears the AI's own TTS →
    (1) mic icon flickers red + STT start/stop beeps, (2) echo transcribed as "user speech" loops the AI
    onto itself until recognition dies, (3) the summary then shows an even 20%×5 split because no real
    turns were captured. Turned OFF to restore stable STT; code stays behind the flag. Re-enable ONLY on a
    physical phone (real AEC) or the emulator with headphones.
  - **RE-ENABLED at client request (`_bargeInEnabled = true`), 2026-07-10.** Client wants voice interrupt
    (no click) + the red mic indicator. **Hard requirement:** headphones on the emulator (or a real phone
    with AEC), otherwise the speaker→mic echo self-interrupts (the bugs above return). **Tap-to-interrupt**
    (`_toggleListening`) remains as a manual fallback that works even with speakers.
  - **OPEN — mic "bip bip" earcon:** the Android `SpeechRecognizer` plays a start/stop earcon; there is no
    Flutter API to disable it. Killing it needs a **native (Kotlin) audio-stream mute** during recognition,
    which is device-specific and risks muting the TTS (both can share `STREAM_MUSIC`). Deferred pending a
    device test (the earcon is often absent/quieter on a real phone than on the emulator).

## LATENCY + ARCH CHANGE (2026-07-10) — "make the conversation fluid"
Measured the real per-turn latency: the bottleneck was **per-message emotion detection** (a full gpt-4o
call BEFORE every reply). Per client's decision, moved emotion extraction to **end-of-call over the whole
transcript** (one call instead of N; cheaper; fluid).
- **chat-stream (`test17.py`):** per-message `_detect_emotion_from_text` **commented out (not deleted)**;
  passes neutral so the reply starts right after moderation. Live-verified: **first word ~0.9–1.4s**
  (was ~2.5–8s) via accurate httpx streaming. (An earlier "4.4s" reading was a urllib client-buffering
  artifact, not real server lag.) The on-call live emotion frame was **dead data** (never shown), so
  nothing visible is lost.
- **Emotions now extracted once at call end:**
  - `_compute_top_emotions_from_transcript()` — one GPT pass (env `EMOTION_MODEL`, default `gpt-4o-mini`)
    over the transcript → 5-category split. Wired into **call-insights** (old `_compute_top_emotions_from_turns`
    kept commented). Preserves Top Emotions + When-feeling-low insights.
  - **Summary** now returns `top_emotions` from the SAME summary GPT call (extended `_build_summary_prompt`
    to also emit the 5 categories; `MoodSummaryResponse.top_emotions`; `dominant_emotion` derived from it).
    Frontend: `CallSummaryResponse.topEmotions` + a new **"Emotions in this chat"** section
    (`_buildEmotionsSection()`) on the summary screen (bars, sorted desc, hidden when empty).
- **Moderation stays per-message** (must block in real-time; ~0.2–0.75s, fast).
- Old per-message emotion code is **commented, not deleted**, per request — for the next refactor pass.
- Live-verified: call-insights + summary both return correct transcript-based emotions (e.g. sad-dominant
  chat → sad 40 / tired 30 / motivated 20 / happy 10).

## AI PERSONA (2026-07-10) — warm wellness companion
Side-effect of the arch change: since `emotion` is now always "neutral", `_build_system_prompt` always
selects the **"neutral"** persona — the emotion-specific templates (happy/sad/angry/anxious/confused in
`_FRIEND_PROMPTS`) are now **unused/dead** (kept for reference, per preserve-not-delete). So the "neutral"
prompt was rewritten (en/de/es) from a casual "close friend / don't sound like a helper" into an
**emotionally-intelligent warm wellness companion**: listens, validates feelings first, celebrates wins,
sits with sadness/stress, gently ties to their day/quests, never a cold assistant, never says AI. One prompt
now adapts to any mood on its own. Note added in `_build_system_prompt`. Server-side only (no app rebuild).
Live-verified: happy turn → celebratory + quest-aware; sad turn → empathetic, sits with them.

## CONFIRMED DECISIONS (2026-07-10)
- **Order:** start with **Summary**, then Profanity, then Barge-in.
- **Profanity block categories:** profanity / harassment / hate / sexual — **EXCLUDE** self-harm /
  emotional distress (those go to a supportive response, never blocked).
- **Profanity warning UX:** visual notice **+** a gentle spoken (TTS) warning; resume listening after.
