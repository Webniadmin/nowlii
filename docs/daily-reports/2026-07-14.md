### Call-duration copy: 10-min → 5-min
- The real call is 5 min (+ one 2.5-min extension). Fixed the stale "10-min" copy in **5 places**:
  `create_quests/enable_card/enable_card.dart`, `edit_quest/enable_card_edit/enable_card_edit.dart`,

### Toggles defaulted to false
- Frontend defaults now match the backend model (`enable_call`/`repeat_quest` = `BooleanField(default=False)`).
  Changed in `create_quests_default.dart` (`enableCall`/`repeatQuest`) and both card widgets'
  internal default + `initialValue` fallback (`enable_card.dart`, `repeat_quest_card.dart`).
- Before this the frontend forced both to `true`, so every created quest silently had them on.

### "Enable call" now does something real
- Decision (client): when on, the quest gets a **"📞 Call Nowlii (5 min)"** button; not a scheduled/auto call.
- `my_quests/today/today.dart` `QuestCard` shows the button **only when `quest.enableCall == true`**;
  tapping pushes `AppRoutespath.aiVoice` with `extra: {'questTitle': quest.task}`.
- `AiVoice` gained an optional `questTitle`; `app_pages.dart` route reads it from `state.extra`;
  `_createAiSession` sends a **delayed (2.5s) opener** so the companion knows the task
  ("I'm starting my task now: <title>…"). Button added to the **Today** tab only.

### "Repeat quest" now does something real
- Decision (client): **materialize copies up-front** (there is no scheduler — no local-notifications,
  no backend cron). In `_createQuest`, after the primary quest, it creates copies for the **next 6
  days (7 days total incl. the selected date)** via the same `QuestService.createQuest`.
- Copies keep `repeat_quest=true` for display but are **not** re-materialized (materialization only
  happens on the create screen, never on the backend), so there's no recursion.

### No past scheduling (new restriction)
- **Finding:** the date picker already limits to today..+6 days (so past *dates* aren't selectable in
  the UI), but the time picker allowed a **past time on "Today"**, and the **backend had zero validation**.
- **Frontend:** `_createQuest` computes the selected date+time and blocks creation if it's before now
  (1-minute grace to absorb the picker's now-default), with a red SnackBar.
- **Backend:** `QuestsSerializers.validate_select_a_date` rejects a past `select_a_date` **on create
  only** (`self.instance is None`), date-level vs `timezone.localdate()` (timezone-safe; edits are not
  blocked; the `add_user_quests` seed command bypasses the serializer, repeat copies are future → all pass).
